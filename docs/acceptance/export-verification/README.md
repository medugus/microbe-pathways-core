# Export Verification Pack

Saved sample outputs and verification checklists for each export format
generated by `src/medugu/logic/exportEngine.ts`. Samples are intentionally
small and illustrative; regenerate from the running app via
**Export section → Copy / Download**.

Last refreshed: **2026-04-20** (browser-phase final regression).

## Verification checklist (per export)

- [ ] Header / meta carries `versions.rule`, `versions.breakpoint`,
      `versions.export`, `versions.build`
- [ ] `fromReleasePackage === true` whenever `accession.release` exists
- [ ] Patient identifiers match accession patient
- [ ] Specimen family + subtype match resolver output
- [ ] Every visible AST row appears as a result with the engine's final SIR
- [ ] Every restricted / suppressed / screening-only row appears as
      **WITHHELD** with a non-empty `suppressionReason`
- [ ] Stewardship and IPC content surface as structured comments/NTE,
      tagged by source
- [ ] No network request fires during export (verify with browser DevTools
      Network tab — filter by the exact moment of clicking Copy/Download)
- [ ] Re-exporting the same released accession yields byte-identical
      payloads

## Production-ready vs placeholder

**Production-ready in this build**

- Visibility/governance contract: WITHHELD masking respects stewardship
  `visibleToClinician` and AST `cascadeDecision`.
- Version pins: every export embeds the four version fields, sourced from
  the frozen `ReleasePackage` when present.
- Client-side guarantee: no `fetch` in the export path; copy uses
  `navigator.clipboard`, download uses an in-page `Blob` + object URL.
- Stable schema: normalised JSON top-level keys (`meta`, `patient`,
  `accession`, `specimen`, `isolates`, `ast`, `stewardship`, `ipc`,
  `validation`, `release`).

**Intentionally placeholder, to harden in Phase 5**

- FHIR `Practitioner` / `PractitionerRole` for the signing pathologist.
- FHIR profile/conformance URLs and terminology-server validation.
- HL7 `ORC` (common order) and `SPM` (specimen) segments.
- HL7 abnormal-flags vocabulary mapping.
- Signing-user identifiers in MSH-9 / OBR-32.
- Amendment / correction status (`DiagnosticReport.status=amended`,
  HL7 result-status `C`) — Phase 5 amendment flow.
- Digital signature / detached cryptographic seal of the
  `ReleasePackage` for non-repudiation.
- Outbound transport (HTTP, MLLP, secure file drop) — the dispatch panel
  is intentionally a mock; see `browser-phase-limitations.md` §3.
